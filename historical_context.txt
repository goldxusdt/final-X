Business Rules and Logic\n\nAll existing business rules remain unchanged, with following additions:\n\n### 4.1 Blog Post Management Rules\n\n#### 4.1.1 Publication Workflow\n\n- New blog posts created with Draft status by default\n- Draft posts not visible on public blog page\n- Publishing post requires all required fields completed and validated\n- Published posts immediately visible on public blog page\n- Editing published post maintains published status unless explicitly unpublished\n- Unpublishing post removes from public blog page immediately\n\n#### 4.1.2 Content Validation\n\n- Post title must be unique (no duplicate titles allowed)\n- Publication date cannot be more than 1 year in future\n- Featured image required before publishing (not required for draft)\n- Content body must contain minimum 50 characters\n- Rich text editor strips potentially harmful HTML/JavaScript\n- All external links in content open in new tab by default\n\n#### 4.1.3 SEO and Metadata\n\n- If SEO meta title not provided, use post title\n- If SEO meta description not provided, use first 160 characters of content\n- Generate URL slug from post title (lowercase, hyphens for spaces)\n- Ensure URL slug uniqueness (append number if duplicate)\n\n#### 4.1.4 Categories and Tags\n\n- Post can belong to multiple categories\n- Post can have unlimited tags\n- Deleting category requires reassigning posts to different category or Uncategorized\n- Deleting tag removes tag from all posts but does not delete posts\n\n### 4.2 Event Management Rules\n\n#### 4.2.1 Publication Workflow\n\n- New events created with Draft status by default\n- Draft events not visible on public events page\n- Publishing event requires all required fields completed and validated\n- Published events immediately visible on public events page\n- Editing published event maintains published status unless explicitly unpublished\n- Unpublishing event removes from public events page immediately\n\n#### 4.2.2 Event Status Logic\n\n- Event automatically categorized as Upcoming if event date/time in future\n- Event automatically categorized as Past if event date/time has passed\n- Past events moved to Past Events section on public page automatically\n- Past events remain editable in admin panel\n- System checks event status on page load and updates display accordingly\n\n#### 4.2.3 Content Validation\n\n- Event title must be unique (no duplicate titles allowed)\n- Event date cannot be in past when creating new event\n- Event date can be in past when editing existing event (for corrections)\n- Featured image required before publishing (not required for draft)\n- Event description must contain minimum 50 characters\n- Registration link must be valid URL format if provided\n- Location field accepts free text (no strict validation)\n\n#### 4.2.4 Registration Link Handling\n\n- If registration link provided, display prominent Register button on detail page\n- Registration link opens in new tab\n- If no registration link, hide Register button\n- Registration link can be updated or removed at any time\n\n### 4.3 Media Management Rules\n\n#### 4.3.1 Upload Rules\n\n- Maximum file size: 5MB per image\n- Accepted formats: JPG, JPEG, PNG, GIF, WebP\n- Reject files exceeding size limit with error message\n- Reject unsupported file formats with error message\n- Automatically optimize images on upload (compress without quality loss)\n- Generate multiple sizes (thumbnail, medium, large) for responsive display\n\n#### 4.3.2 Storage and Organization\n\n- Store images in organized folder structure by upload date\n- Generate unique filename to prevent overwrites\n- Preserve original filename in metadata\n- Store image metadata (alt text, caption, upload date, uploader)\n\n#### 4.3.3 Deletion Rules\n\n- Check if image used in any published blog post or event before deletion\n- If image in use, display warning: This image is used in X posts/events. Deleting will break those references\n- Require confirmation to proceed with deletion\n- If image not in use, delete without warning\n- Permanently remove image files from storage on deletion\n\n### 4.4 Front-End Synchronization Rules\n\n#### 4.4.1 Real-Time Updates\n\n- Publishing blog post or event triggers immediate update to public pages\n- Unpublishing blog post or event triggers immediate removal from public pages\n- Editing published content triggers immediate update on public pages\n- No caching delay for content updates (or cache invalidation on update)\n\n#### 4.4.2 Display Consistency\n\n- Public pages use same rich text rendering as admin preview\n- Featured images display with consistent aspect ratios across all views\n- Categories and tags display with consistent styling\n- Social share buttons use URLs from admin settings (dynamic fetch)\n\n### 4.5 Access Control Rules\n\n#### 4.5.1 Authentication Requirements\n\n- All CMS pages require admin authentication\n- Unauthenticated users redirected to admin login page\n- Session timeout after 15 minutes of inactivity (per existing security policy)\n- Automatic logout and redirect to login page on session expiry\n\n#### 4.5.2 Role-Based Access\n\n- Only users with Admin role can access CMS functionality\n- Regular users cannot access blog/events management pages\n- First registered user automatically assigned Admin role (per existing rule)\n- Admin role assignment managed through existing user management system\n\n### 4.6 Validation and Error Handling\n\n#### 4.6.1 Form Validation\n\n- Validate all required fields before allowing save or publish\n- Display inline error messages next to invalid fields\n- Highlight invalid fields with red border\n- Prevent form submission until all validation passes\n- Display summary of validation errors at top of form\n\n#### 4.6.2 Success and Error Messages\n\n- Display success message after successful post/event creation: Post created successfully\n- Display success message after successful post/event update: Post updated successfully\n- Display success message after successful post/event deletion: Post deleted successfully\n- Display success message after successful publish: Post published successfully\n- Display error message on save failure: Failed to save post. Please try again\n- Display error message on publish failure: Failed to publish post. Please check all required fields\n- Display error message on delete failure: Failed to delete post. Please try again\n- All messages auto-dismiss after 5 seconds or include manual close button\n\n### 4.7 Multi-Language Content Support\n\n#### 4.7.1 Language-Specific Content\n\n- Blog posts and events support multi-language content (per existing multi-language system)\n- Admin can create separate versions of post/event for each supported language\n- Language selector in post/event creation form\n- Public pages display content in user's selected language\n- If content not available in user's language, display default language version\n\n#### 4.7.2 SEO Metadata per Language\n\n- SEO meta title and description can be set per language\n- URL slugs generated per language\n- Locale-specific SEO optimization applied (per existing SEO system)\n\n---\n\n## 5.
